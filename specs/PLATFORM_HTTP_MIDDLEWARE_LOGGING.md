# PLATFORM_HTTP_MIDDLEWARE_LOGGING

Specification for PLATFORM_HTTP_MIDDLEWARE_LOGGING.

## High-Level User POV

Every request that flows through a Codeplane server — whether it originates from the web UI, the CLI, the TUI, an editor integration, the desktop app, or an external automation — passes through the HTTP middleware logging layer. This layer is invisible to end users during normal operation, but it is foundational to every other user-facing quality: reliability, debuggability, performance, and trust.

When something goes wrong — a page loads slowly, an API call returns an unexpected error, a webhook never fires, a workflow run hangs — the middleware logging layer is what gives operators and support engineers the ability to trace exactly what happened. Every request receives a unique identifier that follows it through the entire system. Every response records how long it took, what status it returned, and whether it completed successfully. Operators can tune verbosity to see only errors in production or full request detail during development, without redeploying or restarting the server.

For self-hosting administrators, this means they can point their existing log aggregation tools (Grafana Loki, Elasticsearch, Datadog, or even plain `jq` over stdout) at Codeplane's structured JSON log output and immediately get actionable operational visibility. For the Codeplane development team, it means every bug report can be correlated to a specific request trace, every performance regression can be identified by latency histograms, and every abuse pattern can be detected through rate-limit and error-rate signals.

The logging middleware does not change what users can do with Codeplane. It changes how confidently operators can run it, how quickly problems can be diagnosed, and how reliably the platform can be monitored at scale.

## Acceptance Criteria

### Definition of Done

- [ ] Every HTTP request processed by the Codeplane server emits a structured JSON log line on completion containing the request ID, method, path, status code, response time, and authenticated user context (if present).
- [ ] Every HTTP request processed emits a structured JSON log line on arrival containing the request ID, method, path, and client identity information.
- [ ] Log output is machine-parseable JSON written to stdout by default.
- [ ] Log levels are configurable via the `CODEPLANE_LOG_LEVEL` environment variable with values `debug`, `info`, `warn`, `error`, and `silent`.
- [ ] The default log level in production is `info`.
- [ ] The request ID generated by the `requestId` middleware is present in every log line emitted for that request.
- [ ] Health check endpoints (`/health`, `/healthz`, `/readyz`, `/api/health`) are excluded from request logging at `info` level but included at `debug` level, to prevent log noise from liveness probes.
- [ ] Slow requests (response time exceeding a configurable threshold, default 5000ms) are logged at `warn` level regardless of the configured log level (unless `silent`).
- [ ] Requests that result in 5xx status codes are logged at `error` level regardless of the configured log level (unless `silent`).
- [ ] Requests that result in 4xx status codes are logged at `warn` level.
- [ ] Requests that result in 2xx or 3xx status codes are logged at `info` level.
- [ ] Request and response bodies are never logged at `info` level or below. At `debug` level, request bodies may be logged with automatic redaction of sensitive fields.
- [ ] The following fields are always redacted from any log output: `password`, `token`, `secret`, `authorization`, `cookie`, `codeplane_session`, `private_key`, and any field matching `*_token` or `*_secret` patterns.
- [ ] Log lines include a `service` field set to `"codeplane-server"` for easy filtering in multi-service environments.
- [ ] Log lines include an ISO 8601 `timestamp` field.
- [ ] The middleware does not add more than 1ms of p99 latency overhead per request.
- [ ] The middleware gracefully handles requests where the response stream is aborted or the client disconnects, logging these as `warn` with an `aborted: true` field.
- [ ] The existing Hono built-in `logger()` call is replaced by the new structured logging middleware.
- [ ] All existing `console.log`, `console.warn`, and `console.error` calls in the server application are migrated to use the structured logger so that all server output follows the same format and respects the same log level.

### Edge Cases

- [ ] Requests with extremely long URLs (>8192 characters) must have the logged path truncated to 8192 characters with a `truncated: true` field appended.
- [ ] Requests with missing or malformed `Content-Type` headers must still be logged correctly; the logging middleware must not throw.
- [ ] Requests that trigger the rate limiter (429 responses) must include `rate_limited: true` in the log line.
- [ ] Requests to non-existent routes (404 from the router, not from application logic) must still produce a log line.
- [ ] If the server encounters an unhandled exception during request processing, the error log line must include the error message and a truncated stack trace (first 5 frames).
- [ ] Concurrent requests must not have their log context bleed into each other (request IDs must be isolated per request).
- [ ] The `X-Request-Id` header provided by a client must be validated: only alphanumeric characters, hyphens, and underscores are allowed, maximum 128 characters. Invalid values must be replaced with a server-generated ID and logged at `debug` level.
- [ ] SSE streaming endpoints must log the initial connection at request start and the stream closure at stream end, with total stream duration.
- [ ] The `CODEPLANE_SLOW_REQUEST_THRESHOLD_MS` environment variable must accept integer values between 100 and 60000. Values outside this range must fall back to the default (5000ms) and emit a startup warning.

## Design

### Log Line Schema

Every completed request produces a structured JSON object with the following shape:

**Request arrival log (emitted when request is received):**

```json
{
  "level": "debug",
  "timestamp": "2026-03-21T14:30:00.000Z",
  "service": "codeplane-server",
  "event": "http.request.start",
  "requestId": "m1abc123-4",
  "method": "POST",
  "path": "/api/repos/acme/widgets/issues",
  "query": "page=1&per_page=20",
  "userAgent": "codeplane-cli/0.1.0",
  "ip": "192.168.1.100",
  "contentLength": 256
}
```

**Request completion log (emitted when response is sent):**

```json
{
  "level": "info",
  "timestamp": "2026-03-21T14:30:00.045Z",
  "service": "codeplane-server",
  "event": "http.request.complete",
  "requestId": "m1abc123-4",
  "method": "POST",
  "path": "/api/repos/acme/widgets/issues",
  "status": 201,
  "durationMs": 45,
  "userId": 42,
  "username": "alice",
  "authMethod": "token",
  "responseSize": 512
}
```

**Error log (emitted for 5xx responses or unhandled errors):**

```json
{
  "level": "error",
  "timestamp": "2026-03-21T14:30:00.045Z",
  "service": "codeplane-server",
  "event": "http.request.error",
  "requestId": "m1abc123-4",
  "method": "POST",
  "path": "/api/repos/acme/widgets/issues",
  "status": 500,
  "durationMs": 12,
  "error": "Cannot read properties of undefined (reading 'id')",
  "stack": ["at IssueService.create (...)","at handler (...)","..."],
  "userId": 42,
  "username": "alice",
  "authMethod": "token"
}
```

### SDK Shape

A new structured logger module is added to `packages/sdk/src/lib/logger.ts`:

```
createLogger(options: LoggerOptions): Logger

interface LoggerOptions {
  level: "debug" | "info" | "warn" | "error" | "silent"
  service: string
  pretty?: boolean  // human-readable output for development
}

interface Logger {
  debug(event: string, fields?: Record<string, unknown>): void
  info(event: string, fields?: Record<string, unknown>): void
  warn(event: string, fields?: Record<string, unknown>): void
  error(event: string, fields?: Record<string, unknown>): void
  child(fields: Record<string, unknown>): Logger
}
```

The `child()` method creates a logger with pre-bound fields, enabling request-scoped loggers that automatically include `requestId` on every log call without passing it explicitly.

The `pretty` option, when enabled, outputs colorized, multi-line human-readable logs instead of JSON. This is auto-detected from `NODE_ENV !== "production"` unless explicitly set via `CODEPLANE_LOG_PRETTY=true|false`.

### Middleware Shape

The middleware replaces the current `logger()` from Hono:

```
structuredLogger(logger: Logger): MiddlewareHandler
```

This middleware:
1. Records the request start time via `performance.now()`.
2. At `debug` level, emits an `http.request.start` log line.
3. Calls `await next()`.
4. Computes duration, determines log level from status code, and emits an `http.request.complete` or `http.request.error` log line.
5. Attaches a child logger to the Hono context so downstream route handlers and services can emit structured logs with the request ID pre-bound.

### Sensitive Field Redaction

A redaction utility is included in the logger module:

- Operates on a configurable deny-list of field name patterns.
- Default deny-list: `password`, `token`, `secret`, `authorization`, `cookie`, `session`, `private_key`, `*_token`, `*_secret`, `*_key` (where `*` is a glob prefix).
- Redacted values are replaced with `"[REDACTED]"`.
- Redaction is applied recursively to nested objects up to 3 levels deep.
- Redaction is applied to request body logging at `debug` level only.

### Environment Variable Configuration

| Variable | Type | Default | Description |
|---|---|---|---|
| `CODEPLANE_LOG_LEVEL` | `debug\|info\|warn\|error\|silent` | `info` | Minimum log level emitted |
| `CODEPLANE_LOG_PRETTY` | `true\|false` | auto-detected | Human-readable output mode |
| `CODEPLANE_SLOW_REQUEST_THRESHOLD_MS` | `100–60000` | `5000` | Threshold above which requests are logged as `warn` |
| `CODEPLANE_LOG_HEALTH_CHECKS` | `true\|false` | `false` | Whether to log health check requests at `info` level |

### CLI Command

The `codeplane serve` and `codeplane daemon` commands surface the log level as a flag:

```
codeplane serve --log-level debug
codeplane daemon start --log-level warn
```

This flag overrides the `CODEPLANE_LOG_LEVEL` environment variable for that invocation.

### Admin UI

The Admin panel should surface a read-only "Server Logs" view under Admin → Diagnostics that shows:

- Current configured log level.
- Current slow-request threshold.
- A live-tail of recent structured log lines streamed via SSE (last 500 lines, `warn` and `error` level only).
- A filter bar to search by request ID, path, status code, or username.

This admin surface is a `Gated` feature that can ship as a placeholder initially and be wired up as backend support matures.

### Documentation

The following end-user documentation should be produced:

1. **Self-Hosting Guide: Logging Configuration** — A page in the self-hosting docs explaining all logging environment variables, how to configure log level, how to enable pretty-print mode for development, and how to pipe JSON logs into common log aggregation tools (examples for `jq`, Loki/Promtail, Datadog agent, and Elasticsearch/Filebeat).

2. **Troubleshooting Guide: Using Request IDs** — A page explaining how to find the `X-Request-Id` header in API responses, how to search logs by request ID, and how to include request IDs in bug reports.

3. **CLI Reference Update** — The `codeplane serve` and `codeplane daemon` command references should document the `--log-level` flag.

4. **Admin Guide: Diagnostics** — A page describing the Admin → Diagnostics log viewer, its capabilities, and its limitations (gated feature, recent logs only, not a replacement for log aggregation).

## Permissions & Security

### Authorization

| Surface | Required Role | Notes |
|---|---|---|
| Log output (stdout) | Server operator / sysadmin | Logs are emitted to stdout; access is controlled by OS-level file/process permissions, not Codeplane auth. |
| Admin → Diagnostics log viewer | Admin | Only users with `isAdmin: true` may access the live log tail. |
| `--log-level` CLI flag | Server operator | Whoever starts the server process controls the log level. |
| `CODEPLANE_LOG_LEVEL` env var | Server operator | Same as above; infrastructure-level control. |

### Rate Limiting

- The logging middleware itself does not require rate limiting since it is an internal observability concern.
- The Admin → Diagnostics SSE log stream should be limited to **3 concurrent SSE connections per admin user** to prevent resource exhaustion.

### Data Privacy and PII

- **IP addresses** are logged in request start lines. Operators in GDPR-regulated environments should be informed that IP addresses constitute PII and should configure their log retention and processing accordingly. The documentation must note this.
- **Usernames** are logged in request completion lines. These are already public profile identifiers but should be treated as personal data in log retention policies.
- **Request paths** may contain repository names, issue titles (in query parameters), or other user-generated content. The logging middleware must not attempt to parse or transform these, but documentation should warn operators that log data may contain user-generated strings.
- **Request and response bodies** are never logged at `info` level. At `debug` level, bodies are logged with full sensitive-field redaction applied. The documentation must explicitly warn that `debug` level should not be used in production environments handling sensitive data unless the log pipeline is itself secured.
- **Authorization headers, session cookies, and tokens** are always redacted, at every log level, in every log line.
- **Stack traces** in error logs may expose internal file paths. This is acceptable for operator diagnostics but should not be exposed through any user-facing API.

## Telemetry & Product Analytics

### Business Events

This feature is an infrastructure/platform capability, not a direct user-facing product feature. It does not emit product analytics events in the traditional sense. However, the following operational telemetry events are relevant:

| Event | Properties | Trigger |
|---|---|---|
| `server.started` | `logLevel`, `slowRequestThresholdMs`, `prettyMode`, `version` | Server boot completes |
| `server.log_level_changed` | `previousLevel`, `newLevel`, `source` (env/cli/admin) | Log level changes at runtime (if supported in future) |

### Success Indicators

| Metric | Target | Description |
|---|---|---|
| **Log coverage** | 100% of HTTP requests | Every request that enters the middleware stack produces exactly one completion log line. |
| **Request ID propagation rate** | 100% | Every log line contains a non-empty `requestId` field. |
| **Admin diagnostics adoption** | >50% of admin users view diagnostics within first 30 days | Indicates the log viewer is discoverable and useful. |
| **Mean time to diagnose (MTTD)** | Reduction from baseline | Operators should report faster incident diagnosis after structured logging is available. |
| **Debug mode misuse rate** | <5% of production deployments | Few production instances should be running at `debug` level, as measured by `server.started` telemetry. |

## Observability

### Logging Requirements

The logging middleware is itself the primary logging facility. Its own operational health is observed through:

| What is logged | Level | Structured Context |
|---|---|---|
| Request received | `debug` | `requestId`, `method`, `path`, `query`, `userAgent`, `ip`, `contentLength` |
| Request completed (2xx/3xx) | `info` | `requestId`, `method`, `path`, `status`, `durationMs`, `userId`, `username`, `authMethod`, `responseSize` |
| Request completed (4xx) | `warn` | Same as above plus `error` message from response body if available |
| Request completed (5xx) | `error` | Same as above plus `error`, `stack` (truncated) |
| Slow request | `warn` | Same as completion log plus `slow: true`, `threshold: <configuredMs>` |
| Client disconnect / abort | `warn` | `requestId`, `method`, `path`, `aborted: true`, `durationMs` |
| Rate-limited request | `warn` | Same as completion log plus `rateLimited: true` |
| Health check request | `debug` | Standard completion fields (suppressed at `info` unless `CODEPLANE_LOG_HEALTH_CHECKS=true`) |
| SSE stream opened | `info` | `requestId`, `method`, `path`, `event: "http.sse.open"` |
| SSE stream closed | `info` | `requestId`, `method`, `path`, `event: "http.sse.close"`, `durationMs` |
| Logger initialization | `info` | `logLevel`, `prettyMode`, `slowThreshold`, `service`, at server startup only |
| Invalid client-provided request ID | `debug` | `clientRequestId` (the invalid value), `generatedRequestId` |

### Prometheus Metrics

| Metric | Type | Labels | Description |
|---|---|---|---|
| `codeplane_http_requests_total` | Counter | `method`, `path_template`, `status` | Total HTTP requests processed |
| `codeplane_http_request_duration_seconds` | Histogram | `method`, `path_template`, `status` | Request latency distribution. Buckets: 0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10 |

### Alerts

#### Alert: High 5xx Error Rate

- **Condition**: `rate(codeplane_http_requests_total{status=~"5.."}[5m]) / rate(codeplane_http_requests_total[5m]) > 0.05`
- **Severity**: Critical
- **Summary**: More than 5% of HTTP requests are returning 5xx errors over a 5-minute window.

**Runbook:**
1. Check the Codeplane server logs filtered for `level: "error"` and `event: "http.request.error"` in the last 15 minutes.
2. Identify the most common `path` and `error` message in error logs. Determine if the errors are concentrated on a single route or spread across many.
3. If concentrated on a single route, check if a recent deployment introduced a regression. Roll back if confirmed.
4. If spread across many routes, check database connectivity (`/health` endpoint response), memory pressure, and disk I/O on the server host.
5. If the database is unreachable, follow the database recovery runbook.
6. If memory is exhausted, restart the server process and investigate for memory leaks using heap snapshots.

#### Alert: High Request Latency (p95)

- **Condition**: `histogram_quantile(0.95, rate(codeplane_http_request_duration_seconds_bucket[5m])) > 2`
- **Severity**: Warning
- **Summary**: The 95th percentile request latency exceeds 2 seconds over a 5-minute window.

**Runbook:**
1. Check structured logs for requests with `slow: true` to identify which routes are slow.
2. Determine if slowness is concentrated on specific route families (e.g., search, workflow runs, repository operations).
3. Check database query performance: look for long-running queries in the database's `pg_stat_activity` or equivalent.
4. Check if the server is under unusually high load: review `codeplane_http_requests_in_flight` and `codeplane_http_requests_total` rate.
5. If a specific route is slow, review recent code changes to that route's service layer.
6. If general slowness, check host CPU, memory, disk I/O, and network latency to the database.

#### Alert: High Rate-Limit Hit Rate

- **Condition**: `rate(codeplane_http_requests_total{status="429"}[5m]) > 10`
- **Severity**: Warning
- **Summary**: More than 10 rate-limited requests per second sustained over 5 minutes, indicating possible abuse or a misconfigured client.

**Runbook:**
1. Check structured logs for requests with `rateLimited: true` to identify the source IP or user ID.
2. Determine if this is a single abusive client or a legitimate client with a misconfigured polling interval.
3. If a single abusive source, consider IP-level blocking at the load balancer or firewall level.
4. If a legitimate client, notify the user/team about the rate limit and suggest they reduce request frequency or use SSE streaming endpoints where available.
5. If the rate limit itself is too low for legitimate traffic patterns, consider increasing `rateLimit()` from 120 to a higher value and monitoring.

#### Alert: Elevated Client Abort Rate

- **Condition**: `rate(codeplane_http_aborted_requests_total[5m]) / rate(codeplane_http_requests_total[5m]) > 0.1`
- **Severity**: Warning
- **Summary**: More than 10% of requests are being aborted by clients, indicating timeouts or UX issues.

**Runbook:**
1. Check structured logs for `aborted: true` to identify which routes are most affected.
2. Cross-reference with `codeplane_http_request_duration_seconds` to determine if clients are timing out due to slow responses.
3. If abort rate correlates with high latency, follow the High Request Latency runbook.
4. If latency is normal but aborts are high, investigate client-side behavior: check if the web UI, CLI, or TUI has a misconfigured timeout or is navigating away from pages before requests complete.
5. If concentrated on SSE endpoints, check if proxy/load balancer timeouts are misconfigured for long-lived connections.

### Error Cases and Failure Modes

| Error Case | Behavior | Log Level |
|---|---|---|
| Logger fails to write to stdout (disk full, pipe broken) | Middleware continues processing the request; log line is silently dropped. Server does not crash. | N/A (cannot log) |
| `CODEPLANE_LOG_LEVEL` set to invalid value | Falls back to `info`, emits a startup warning to stderr. | `warn` at startup |
| `CODEPLANE_SLOW_REQUEST_THRESHOLD_MS` set to invalid value | Falls back to 5000ms, emits a startup warning. | `warn` at startup |
| Unhandled exception in logging middleware itself | Caught by a try/catch wrapper; request processing continues. Error is written to stderr as a fallback. | `error` to stderr |
| Request body exceeds memory limits during debug-level body logging | Body logging is skipped for that request; a `bodyLoggingSkipped: true` field is added to the completion log. | `debug` |
| Extremely high request volume causes log backpressure | Logs are written synchronously to stdout (Bun's default). If stdout buffer is full, writes may block briefly. This is acceptable and expected. | N/A |

## Verification

### API Integration Tests

- [ ] `logs a structured JSON line for a successful GET request` — Send `GET /api/health`, capture stdout, parse JSON, verify `event`, `method`, `path`, `status: 200`, `durationMs >= 0`, `requestId` is present.
- [ ] `logs a structured JSON line for a successful POST request` — Send `POST /api/repos/:owner/:repo/issues` with valid body, verify completion log has `status: 201`, `userId`, `username`.
- [ ] `logs at error level for 500 responses` — Trigger a known 500-producing endpoint (or mock), verify log line has `level: "error"`, `error` field present, `stack` array present.
- [ ] `logs at warn level for 4xx responses` — Send a request producing a 400, verify `level: "warn"`.
- [ ] `logs at warn level for 404 responses` — Request a non-existent route, verify `level: "warn"`, `status: 404`.
- [ ] `logs at warn level for rate-limited requests` — Exceed the rate limit, verify log line has `level: "warn"`, `status: 429`, `rateLimited: true`.
- [ ] `includes X-Request-Id in response header and log` — Send a request, read the `X-Request-Id` response header, find the matching `requestId` in the log line.
- [ ] `preserves valid client-provided X-Request-Id` — Send a request with `X-Request-Id: my-valid-id-123`, verify log line uses `requestId: "my-valid-id-123"`.
- [ ] `rejects invalid client-provided X-Request-Id` — Send `X-Request-Id: "invalid id with spaces!!!"`, verify log line has a server-generated `requestId` (not the client value).
- [ ] `rejects overly long client-provided X-Request-Id` — Send `X-Request-Id` with 200 characters, verify server-generated ID is used instead.
- [ ] `excludes health check requests at info level` — Set log level to `info`, send `GET /health`, verify no log line is emitted for that request.
- [ ] `includes health check requests at debug level` — Set log level to `debug`, send `GET /health`, verify a log line is emitted.
- [ ] `includes health check requests when CODEPLANE_LOG_HEALTH_CHECKS=true` — Set env var, send `GET /health` at `info` level, verify log is emitted.
- [ ] `marks slow requests with slow: true` — Set threshold to 100ms, send a request to a route known to take >100ms, verify `slow: true` and `level: "warn"`.
- [ ] `does not mark fast requests as slow` — Set threshold to 60000ms, send `GET /health`, verify no `slow` field.
- [ ] `redacts Authorization header in debug body logging` — Set level to `debug`, send a request, verify no raw token appears in any log line.
- [ ] `redacts password fields in debug body logging` — Set level to `debug`, send a POST with `{"password": "secret123"}`, verify the logged body shows `"password": "[REDACTED]"`.
- [ ] `redacts nested sensitive fields` — Send `{"user": {"token": "abc123"}}` at debug level, verify nested redaction.
- [ ] `never logs request bodies at info level` — Set level to `info`, send a POST with a body, verify no body content in any log line.
- [ ] `truncates extremely long paths in log output` — Send a request with a path >8192 characters, verify the logged `path` is truncated and `truncated: true` is present.
- [ ] `logs correct duration for requests` — Send a request, verify `durationMs` is a non-negative number and is roughly consistent with observed wall-clock time (±50ms tolerance).
- [ ] `logs response size` — Send a request, verify `responseSize` matches the actual `Content-Length` of the response.
- [ ] `does not emit logs at silent level` — Set level to `silent`, send multiple requests, verify no log lines are emitted.
- [ ] `handles concurrent requests with isolated request IDs` — Send 50 concurrent requests, collect all log lines, verify each `requestId` appears exactly once in completion logs and no two requests share an ID.
- [ ] `logs SSE stream open and close` — Open an SSE connection (e.g., notification stream), verify `http.sse.open` log, close the connection, verify `http.sse.close` log with `durationMs`.
- [ ] `logs unauthenticated requests without userId` — Send a request without auth, verify log line has no `userId` or `username` fields (or they are null).
- [ ] `logs authenticated token requests with authMethod: token` — Send a request with a PAT, verify `authMethod: "token"`.
- [ ] `logs authenticated session requests with authMethod: session` — Send a request with a session cookie, verify `authMethod: "session"`.
- [ ] `respects CODEPLANE_LOG_LEVEL=warn` — Set level to `warn`, send a successful request, verify no `info`-level log is emitted. Send a request producing a 4xx, verify `warn` log is emitted.
- [ ] `respects CODEPLANE_LOG_LEVEL=error` — Set level to `error`, send a 4xx request, verify no log. Trigger a 5xx, verify error log is emitted.
- [ ] `falls back to info for invalid CODEPLANE_LOG_LEVEL` — Set `CODEPLANE_LOG_LEVEL=invalid`, start server, verify it uses `info` level and emits a startup warning.
- [ ] `falls back to default for out-of-range slow threshold` — Set `CODEPLANE_SLOW_REQUEST_THRESHOLD_MS=50` (below 100), verify fallback to 5000ms.
- [ ] `falls back to default for non-numeric slow threshold` — Set `CODEPLANE_SLOW_REQUEST_THRESHOLD_MS=abc`, verify fallback to 5000ms.
- [ ] `handles maximum valid slow threshold` — Set `CODEPLANE_SLOW_REQUEST_THRESHOLD_MS=60000`, verify it is accepted (no fallback warning).
- [ ] `handles minimum valid slow threshold` — Set `CODEPLANE_SLOW_REQUEST_THRESHOLD_MS=100`, verify it is accepted.
