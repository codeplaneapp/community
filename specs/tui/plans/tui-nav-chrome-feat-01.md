# Implementation Plan for TUI_SCREEN_ROUTER (tui-nav-chrome-feat-01)

## 1. Overview
This plan details the process to transform the minimal `ScreenRouter` and `GlobalKeybindings` into a robust, stack-based navigation system for the Codeplane TUI. This includes handling loading states, mid-session authentication expirations, go-to mode with visual indicators and timeouts, and integration with the telemetry and error boundary systems.

## 2. Step-by-Step Implementation

### Step 1: Create Loading and Auth Expired Screens
**Target Files:**
- `apps/tui/src/components/LoadingScreen.tsx` (New File)
- `apps/tui/src/components/AuthExpiredScreen.tsx` (New File)

**Actions:**
1. Create `LoadingScreen.tsx` utilizing OpenTUI's `<box>` and `<text>`. Consume `useLoading()` to acquire the `spinnerFrame` animation, displaying it alongside an optional `label` prop. Center the content.
2. Create `AuthExpiredScreen.tsx` designed to handle mid-session 401s. It should display a `Session expired` message referencing the `host`, provide instructions to run `codeplane auth login`, and show keyboard hints (`R retry | q quit`).

### Step 2: Enhance LoadingProvider & Initial Stack Deep Links
**Target Files:**
- `apps/tui/src/providers/LoadingProvider.tsx`
- `apps/tui/src/index.tsx`

**Actions:**
1. Modify `LoadingProvider.tsx` to add an `initialStatusBarError` prop which sets the default state of `statusBarError` on mount.
2. Expose a `setStatusBarError(message: string, durationMs?: number)` method in the `LoadingContextType` to allow dynamic transient error messages (such as go-to context failures). Clear the error via `setTimeout` based on `durationMs`.
3. In `index.tsx`, when `buildInitialStack()` is called, capture `deepLinkResult.error` and pass it into `LoadingProvider` as the `initialStatusBarError` prop.

### Step 3: Implement Go-To Status Bar Indicator
**Target File:** `apps/tui/src/components/StatusBar.tsx`

**Actions:**
1. Locate the loop where `sortedHints` are rendered into the status bar.
2. Add a conditional check: `if (hint.keys === "-- GO TO --")`.
3. Render the matched string using `<text fg={theme.warning} bold>`. This correctly highlights the indicator generated by `GlobalKeybindings` when go-to mode is active.

### Step 4: Wire Navigation Telemetry
**Target File:** `apps/tui/src/providers/NavigationProvider.tsx`

**Actions:**
1. Import `emit` from `../lib/telemetry.js`.
2. Inside the `push` function, queue a microtask following the state update to call `emit("tui.navigate.push", { from_screen, to_screen, stack_depth, has_repo_context })`.
3. Inside the `pop` function, similarly queue a microtask calling `emit("tui.navigate.pop", { from_screen, to_screen, stack_depth })`.

### Step 5: Complete GlobalKeybindings implementation
**Target File:** `apps/tui/src/components/GlobalKeybindings.tsx`

**Actions:**
1. Perform a comprehensive rewrite based on the spec.
2. Manage state/refs for `goToActive`, timeouts (1500ms), and keybinding scopes.
3. Implement `enterGoTo()`: register a `PRIORITY.GOTO` scope with all go-to destination keys, `escape`, and `q`. Furthermore, add fallbacks for other lowercase letters (`a-z`) to silently cancel the mode on invalid keypresses.
4. Apply `statusBarCtx.overrideHints` to swap normal hints with go-to mapping labels, and inject `{ keys: "-- GO TO --", label: "", order: -1 }` to trigger the status bar highlight.
5. Implement `handleGoToKey(key)`: clear the timeout/scope, evaluate context availability via `executeGoTo()`. If it fails, invoke `loading.setStatusBarError("No repository in context", 2000)` and emit `tui.navigate.goto_fail`. If successful, emit `tui.navigate.goto`.
6. Wire up `onHelp` and `onCommandPalette` to interface with `OverlayManager`.

### Step 6: Refactor ScreenRouter
**Target File:** `apps/tui/src/router/ScreenRouter.tsx`

**Actions:**
1. Ingest `useLoading()`, `useAuth()`, and `useScreenKeybindings()`.
2. Track `currentScreenLoading` for an auth-related error type or `401` HTTP status. If present, immediately render `AuthExpiredScreen`.
3. Support an `R` retry keybinding inside the router that targets `auth.retry()` (if expired) or `loading.retryCallback()`.
4. Provide the `LoadingScreen` component when `isScreenLoading` is true.
5. Guarantee complete unmount/mount cycles for top-of-stack changes by injecting `key={nav.currentScreen.id}` onto the resolved component.

### Step 7: Apply Extensive E2E Test Suite
**Target File:** `e2e/tui/app-shell.test.ts`

**Actions:**
1. Embed the 30+ tests provided in the Engineering Spec into the existing `app-shell.test.ts` file.
2. Segment into proper `describe` blocks (`TUI_SCREEN_ROUTER — Snapshots`, `Keyboard interactions`, `Deep links`, `Responsive`, `Integration`).
3. Maintain tests that fail due to unimplemented backends, as strict adherence to the spec implies no skipping or commenting out of unimplemented feature dependencies.

## 3. Review and Verification
Ensure there are no infinite React renders by maintaining correct ref/effect usage in `GlobalKeybindings.tsx`. Validate text input captures work implicitly within `OpenTUI` without overriding global bindings by firing test scripts against inputs and verifying they block 'g' or 'q'. Confirm the stack correctly handles multi-rapid `q` presses natively due to React 19 functional state update batching.
