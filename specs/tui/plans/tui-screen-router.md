# TUI_SCREEN_ROUTER — Implementation Plan

This implementation plan covers the creation of the `ScreenRouter` component, the `ScreenErrorBoundary`, and their integration into the Codeplane TUI `AppShell` as defined by the `TUI_SCREEN_ROUTER` engineering spec.

## Step 1: Create `ScreenErrorBoundary`
**File:** `apps/tui/src/router/ScreenErrorBoundary.tsx`

Create a new per-screen error boundary component that isolates crashes and provides recovery options without tearing down the entire terminal shell.
1. Define a class component `ScreenErrorBoundary` extending `React.Component`.
2. Add state for `hasError`, `error`, `showStack`, and `retryToken`.
3. Implement `getDerivedStateFromError` to catch exceptions.
4. Implement `componentDidCatch` to log the error safely via `process.stderr.write` (to avoid corrupting the TUI render output).
5. Render a fallback UI using OpenTUI `<box>` and `<text>` components if `hasError` is true. Display the error message, the stack trace (if toggled), and standard recovery actions: `q` to go back, `r` to retry, and `s` to toggle stack trace.
6. Add an inner functional component (e.g. `<ScreenErrorActions>`) that utilizes `useScreenKeybindings` to bind `q`, `r`, and `s` respectively.
7. Wrap the normal `children` inside `<React.Fragment key={this.state.retryToken}>` so retrying correctly forces a remount.

## Step 2: Implement `ScreenRouter`
**File:** `apps/tui/src/router/ScreenRouter.tsx`

Rewrite the existing placeholder router to be robust, key-isolated, and structurally safe.
1. Import `React`, `useNavigation` from `../providers/NavigationProvider.js`, `ScreenErrorBoundary`, and `DEFAULT_ROOT_SCREEN` from `./types.js`.
2. Destructure `currentScreen`, `stack`, and `push` from the `useNavigation()` hook.
3. Implement the **Empty Stack Guard**: If `!currentScreen` or `stack.length === 0`, return a new component `<EmptyStackGuard push={push} />`.
4. Define `EmptyStackGuard` to perform an effect-based redirect (`push(DEFAULT_ROOT_SCREEN)`) inside a `React.useEffect` to abide by React's rendering lifecycle, returning a simple loading `<box>`.
5. Enhance the unknown screen fallback to display `<text color="gray">Press q to go back.</text>` alongside the error.
6. Retrieve the resolved `Component` from the registry and prepare its `props` (`entry` and `params`).
7. **CRITICAL:** Return `<Component {...props} />` wrapped inside `<ScreenErrorBoundary>`.
8. **CRITICAL:** Provide `key={currentScreen.id}` on the error boundary wrapper. This forces a complete React unmount/remount on navigation updates, fully isolating screen instances.

## Step 3: Update `AppShell` Integration
**File:** `apps/tui/src/components/AppShell.tsx`

Explicitly embed the router into the application shell.
1. Remove the `children` parameter and prop from `AppShell`.
2. Import `ScreenRouter` from `../router/ScreenRouter.js`.
3. Render `<ScreenRouter />` securely inside the `<box flexGrow={1} width="100%">` element, situated between `HeaderBar` and `StatusBar`.

## Step 4: Simplify TUI Bootstrap
**File:** `apps/tui/src/index.tsx`

Update the provider stack in the main entry point to reflect the changes to `AppShell`.
1. Remove the `ScreenRouter` import from `index.tsx`.
2. Remove the `<ScreenRouter />` tag that is currently nested inside `<AppShell>`.
3. Leave `<AppShell />` as a self-closing element inside the `<GlobalKeybindings>` wrapper.

## Step 5: Update `PlaceholderScreen`
**File:** `apps/tui/src/screens/PlaceholderScreen.tsx`

Simplify the dummy placeholder for missing screens to match the final aesthetic constraints.
1. Change the outer container to `<box flexDirection="column" justifyContent="center" alignItems="center" flexGrow={1}>`.
2. Remove the old param-listing logic.
3. Display `<text color="gray">{entry.breadcrumb}</text>` followed by `<text color="gray" dim>Screen not yet implemented</text>`.

## Step 6: Expand Router Exports
**File:** `apps/tui/src/router/index.ts`

Ensure proper module visibility for the newly constructed boundary.
1. Add the explicit export: `export { ScreenErrorBoundary } from "./ScreenErrorBoundary.js";`.
2. Verify `ScreenRouter`, `screenRegistry`, and types (`ScreenName`, `MAX_STACK_DEPTH`, `DEFAULT_ROOT_SCREEN`, `ScreenEntry`, `NavigationContext`, `ScreenDefinition`, `ScreenComponentProps`) are all exported.

## Step 7: Add Router E2E Tests
**File:** `e2e/tui/app-shell.test.ts`

Include comprehensive tests verifying `TUI_SCREEN_ROUTER` behavioral requirements using the `@microsoft/tui-test` framework and snapshot validation.
1. **Snapshot Tests:** Add tests simulating an initial launch (showing Dashboard), breadcrumbs generated by `g r` (Dashboard > Repositories), and deep-stack breadcrumbs passing `--screen issues --repo owner/repo`.
2. **Responsive Rendering:** Write tests verifying 80x24 breadcrumb truncation layout, "Terminal too small" warnings at 60x20, and correct runtime resizes during navigation.
3. **Keyboard & Go-To Mode Validation:** Test that `g` activates go-to prompt, invalid inputs throw contextual errors, `Esc` dismisses overlays properly, and `q` pops up the navigation tree up to an eventual exit.
4. **Deep Linking Tests:** Construct tests asserting that bad `--screen` requests and missing required `--repo` arguments trigger graceful fallbacks directly to the Dashboard containing appropriate error context.
5. **Expected Failures:** Append the `router-auth-expiry-mid-session` and `router-loading-state` tests but leave them unmocked and failing per project philosophy until their associated data APIs are implemented.
