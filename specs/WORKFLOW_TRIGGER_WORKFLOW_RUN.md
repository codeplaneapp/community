# WORKFLOW_TRIGGER_WORKFLOW_RUN

Specification for WORKFLOW_TRIGGER_WORKFLOW_RUN.

## High-Level User POV

The WORKFLOW_TRIGGER_WORKFLOW_RUN trigger enables multi-stage pipeline chaining where completion, success, or failure of one workflow automatically dispatches dependent workflows. Users configure triggers using `on: { workflow_run: { workflows: [...], types: [...], branches: [...] } }` syntax. Key scenarios include: (1) Multi-stage pipelines where a build workflow triggers deploy on success, (2) Failure notification workflows that fire when critical pipelines fail, (3) Glob-pattern matching across workflow families (e.g., `build-*` triggers a unified integration test suite), (4) Branch-filtered triggers so staging deploys only chain from staging branch runs. The trigger carries the originating run's metadata (run ID, workflow name, conclusion, head change ID, branch) as context available to the triggered workflow. Users can inspect the chain in the web UI via a visual parent→child run lineage graph, in the CLI via `codeplane workflow runs --triggered-by <run-id>`, and in the TUI via a dedicated chain view.

## Acceptance Criteria

Definition of Done: (1) Workflow definitions accept `on.workflow_run` trigger config with `workflows`, `types`, and optional `branches` fields. (2) When a workflow run completes, the trigger evaluation engine matches all workflow definitions with compatible `workflow_run` triggers. (3) `types` supports `completed`, `success`, `failure`, `cancelled`, and `requested` (dispatch-time). (4) `workflows` supports exact names and glob patterns via micromatch. (5) `branches` supports exact and glob patterns against the head bookmark of the originating run. (6) Matched workflows are dispatched as new runs with `trigger_type: 'workflow_run'` and `triggered_by_run_id` foreign key. (7) The triggered run receives `context.workflow_run` payload containing originating run metadata. (8) Chain depth is bounded to 5 levels; exceeding it logs a warning and skips dispatch. (9) Self-trigger exclusion: a workflow cannot trigger itself to prevent infinite loops. (10) Circular chain detection across distinct workflow names within a chain. (11) API exposes `GET /repos/:owner/:repo/workflow-runs/:id/triggered-runs` and `GET /repos/:owner/:repo/workflow-runs/:id/trigger-chain`. (12) Web UI shows trigger chain lineage on run detail page. (13) CLI `workflow runs` supports `--triggered-by` filter. (14) TUI run detail screen shows chain context. (15) SDK `WorkflowBuilder.on.workflowRun()` builder compiles to correct trigger config. (16) All trigger evaluations are idempotent—reprocessing a completion event does not create duplicate runs. (17) Feature flag `workflow_trigger_workflow_run` gates the capability.

Edge Cases: (1) Originating workflow is deleted before completion—trigger still fires using snapshot name. (2) Multiple workflows match the same completion—all dispatch concurrently. (3) Glob pattern `*` does not match the workflow's own name (self-exclusion). (4) Branch field absent on originating run—branch filters treat as non-matching. (5) Concurrent completions of the same workflow—each independently evaluated. (6) Triggered run itself fails—its own completion re-enters trigger evaluation respecting chain depth. (7) Workflow renamed between trigger config and run completion—matches against name at completion time. (8) Empty `workflows` array—trigger never matches (validated at definition parse time). (9) `types: []` defaults to `['completed']`. (10) Run cancelled mid-execution—fires `cancelled` type triggers.

Boundary Constraints: Max chain depth 5. Max glob patterns per trigger 20. Max workflows per trigger 50. Max concurrent triggered dispatches per completion event 10. Trigger evaluation timeout 5s. Max `workflow_run` triggers per repository 100. Rate limit 60 triggered dispatches per repo per minute. Minimum dispatch delay 100ms (debounce).

## Design

TypeScript SDK: `WorkflowBuilder.on.workflowRun({ workflows: string[], types?: WorkflowRunTriggerType[], branches?: string[] })` compiles to `{ on: { workflow_run: { workflows, types, branches } } }`. Types union: `'completed' | 'success' | 'failure' | 'cancelled' | 'requested'`.

JSON trigger config shape: `{ on: { workflow_run: { workflows: string[], types?: string[], branches?: string[] } } }`.

Server implementation: (1) `packages/sdk/src/services/workflow/triggers/workflow-run.ts` — `evaluateWorkflowRunTrigger(completedRun, triggerConfig): boolean` using micromatch for glob matching. (2) `packages/sdk/src/services/workflow/trigger-evaluation.ts` — extended to call workflow_run evaluation after run state transitions. (3) `packages/sdk/src/services/workflow/chain.ts` — chain depth tracking, self-trigger exclusion, circular detection. (4) New DB columns: `workflow_runs.triggered_by_run_id` (nullable FK), `workflow_runs.chain_depth` (integer, default 0).

API routes: `POST /repos/:owner/:repo/workflows/:id/dispatch` extended with `trigger_type` and `triggered_by_run_id`. `GET /repos/:owner/:repo/workflow-runs/:id/triggered-runs` returns child runs. `GET /repos/:owner/:repo/workflow-runs/:id/trigger-chain` returns full ancestor/descendant chain.

Web UI: Run detail page gains a 'Trigger Chain' tab showing DAG of parent→child runs with status badges. Workflow definition editor gains `workflow_run` trigger config UI with autocomplete for workflow names and type checkboxes.

CLI: `codeplane workflow runs --triggered-by <run-id>` lists triggered children. `codeplane workflow runs <id> --chain` shows full chain. `codeplane workflow runs <id>` detail output includes `triggered_by` field.

TUI: Run detail screen shows chain context panel. Run list supports filter by trigger type.

Editor integrations: VS Code and Neovim show trigger chain in workflow run detail views via existing webview/picker patterns.

Documentation: (1) Workflow trigger reference page updated with `workflow_run` section. (2) Multi-stage pipeline tutorial. (3) Chain depth and loop prevention explanation. (4) Migration guide for existing polling-based chaining patterns. (5) SDK builder API reference for `on.workflowRun()`.

## Permissions & Security

Trigger dispatch is system-initiated—no user identity is impersonated. The triggered run inherits the repository context and runs with workflow execution permissions (same as manually dispatched runs). Self-trigger exclusion is enforced at the engine level, not via permissions. Chain depth bound (max 5) acts as a resource abuse prevention mechanism. Rate limiting (60 triggered dispatches/repo/minute) prevents runaway fan-out. Repository admin permission is required to configure `workflow_run` triggers in workflow definitions. Read permission on workflow runs is required to view trigger chain APIs. The `triggered_by_run_id` FK is always set by the system, never accepted from external API input. Feature flag `workflow_trigger_workflow_run` must be enabled for trigger evaluation to proceed; if disabled, `workflow_run` triggers are silently skipped during evaluation. Webhook delivery for triggered runs includes the `triggered_by` context but redacts any secrets from the originating run's environment.

## Telemetry & Product Analytics

Business Events: (1) `workflow_run.trigger_evaluated` — emitted per completion event, properties: `repo_id`, `completed_run_id`, `matched_count`, `skipped_self_trigger`, `evaluation_duration_ms`. (2) `workflow_run.trigger_dispatched` — emitted per triggered dispatch, properties: `repo_id`, `triggered_run_id`, `triggering_run_id`, `chain_depth`, `trigger_type_matched`, `workflow_name`. (3) `workflow_run.chain_depth_exceeded` — emitted when dispatch is skipped due to depth limit, properties: `repo_id`, `run_id`, `chain_depth`, `would_trigger_workflow`. (4) `workflow_run.circular_chain_detected` — emitted when circular reference detected, properties: `repo_id`, `run_id`, `chain_workflow_names`.

Funnel Metrics: (1) Workflow completions → trigger evaluations → matched triggers → dispatched runs → successful triggered runs. (2) Trigger config adoption: repos with ≥1 workflow_run trigger / total repos with workflows. (3) Average chain depth across all triggered runs. (4) P50/P95/P99 trigger evaluation latency. (5) Triggered run success rate vs manually dispatched run success rate. (6) Fan-out distribution: histogram of triggered runs per completion event. (7) Rate limit hit frequency per repo.

## Observability

Structured Log Points: (1) `workflow.trigger.evaluation.start` — level info, run_id, workflow_count_to_evaluate. (2) `workflow.trigger.evaluation.match` — level info, run_id, matched_workflow_id, trigger_config. (3) `workflow.trigger.evaluation.no_match` — level debug, run_id. (4) `workflow.trigger.dispatch.start` — level info, triggering_run_id, target_workflow_id, chain_depth. (5) `workflow.trigger.dispatch.success` — level info, triggered_run_id. (6) `workflow.trigger.dispatch.error` — level error, triggering_run_id, error_message. (7) `workflow.trigger.self_excluded` — level warn, run_id, workflow_name. (8) `workflow.trigger.chain_depth_exceeded` — level warn, run_id, chain_depth, max_depth. (9) `workflow.trigger.circular_detected` — level error, run_id, chain. (10) `workflow.trigger.rate_limited` — level warn, repo_id, current_rate. (11) `workflow.trigger.evaluation.timeout` — level error, run_id, duration_ms.

Prometheus Metrics: (1) `codeplane_workflow_trigger_evaluations_total` counter, labels: repo_id, result(matched|no_match|error). (2) `codeplane_workflow_trigger_dispatches_total` counter, labels: repo_id, trigger_type, chain_depth. (3) `codeplane_workflow_trigger_evaluation_duration_seconds` histogram. (4) `codeplane_workflow_trigger_chain_depth` histogram. (5) `codeplane_workflow_trigger_rate_limited_total` counter, labels: repo_id. (6) `codeplane_workflow_trigger_errors_total` counter, labels: repo_id, error_type.

Alerts: (1) `WorkflowTriggerEvaluationLatencyHigh` — P99 > 3s for 5m, runbook: check DB query performance on workflow definitions table, verify micromatch glob complexity, check for repos with excessive trigger configs. (2) `WorkflowTriggerChainDepthExceededRate` — >10 depth-exceeded events in 5m, runbook: identify repos hitting limits, review workflow configs for unintentional deep chains, consider notifying repo admins. (3) `WorkflowTriggerDispatchErrorRate` — error rate >5% for 5m, runbook: check workflow service health, verify DB connectivity, check for schema migration issues on triggered_by_run_id column. (4) `WorkflowTriggerRateLimitHitRate` — >50 rate limit events in 10m, runbook: identify repos causing fan-out, review glob patterns for overly broad matches, consider per-repo limit adjustments.

Error/Failure Modes: (1) DB unavailable during trigger evaluation — log error, skip dispatch, rely on idempotent reprocessing. (2) Glob pattern compilation error — log error, skip that trigger config, continue evaluating others. (3) Triggered run creation fails — log error, emit dispatch.error event, do not retry automatically. (4) Chain depth check fails (DB read error) — treat as depth exceeded, skip dispatch (fail-safe). (5) Rate limit storage unavailable — allow dispatch (fail-open) but log warning. (6) Concurrent completion events for same run — idempotency key on (triggering_run_id, target_workflow_id) prevents duplicates. (7) Feature flag check fails — treat as disabled (fail-safe). (8) Trigger evaluation timeout — abort evaluation, log timeout, emit metric. (9) Originating run record deleted before evaluation completes — use cached run metadata from completion event payload.
