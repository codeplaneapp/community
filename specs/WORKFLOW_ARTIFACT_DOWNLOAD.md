# WORKFLOW_ARTIFACT_DOWNLOAD

Specification for WORKFLOW_ARTIFACT_DOWNLOAD.

## High-Level User POV

When a workflow run completes in Codeplane, it often produces build artifacts—compiled binaries, test reports, coverage files, bundled archives, container images, or any file the workflow explicitly publishes. The Workflow Artifact Download feature lets any authorized user retrieve those artifacts from any Codeplane surface: the web UI, CLI, TUI, API, or editor integrations.

A developer navigates to a completed workflow run, sees a list of artifacts the run produced, and clicks or invokes a download on the one they need. The system generates a short-lived, signed download URL that streams the artifact's raw bytes to the user's browser or local filesystem. The download is direct and fast—there is no intermediate packaging step, no zip wrapping, and no re-encoding. Whatever the workflow uploaded is exactly what the user receives, byte-for-byte, with the original content type preserved.

From the CLI, `codeplane artifact download <runId> <name>` fetches the artifact metadata, follows the signed download URL, and writes the file to disk. The user can specify a custom output path or let it default to the artifact's original name. From the TUI, pressing `D` on a focused artifact row delegates to the same CLI flow and shows progress in the status bar. From the web UI, clicking the download button on an artifact row or detail panel triggers a browser download using the signed URL. In every case, the experience is: see the artifact, download the artifact, use the artifact.

Artifacts that are still pending upload confirmation cannot be downloaded—the user is told the artifact is not yet ready. Artifacts that have expired are shown but cannot be downloaded—the user is told the artifact has expired and the underlying data has been cleaned up. These states are always visible so the user understands why a download may not be available without needing to investigate further.

This feature closes the loop on Codeplane's workflow automation story: workflows produce artifacts, and developers consume them seamlessly across every client surface.

## Acceptance Criteria

### Definition of Done

- [ ] The API endpoint `GET /api/repos/:owner/:repo/actions/runs/:id/artifacts/:name/download` returns artifact metadata including a signed `download_url` for artifacts in `ready` status
- [ ] The signed download URL is generated by the blob store with a 5-minute expiry and HMAC-SHA256 token validation
- [ ] The blob download endpoint `GET /api/blobs/download/:key` streams the raw artifact bytes with correct `Content-Type` and `Content-Disposition` headers
- [ ] The CLI `artifact download` command fetches the metadata, follows the `download_url`, and streams the content to a local file
- [ ] The TUI download action delegates to the CLI `artifact download` command and surfaces progress/result in the status bar
- [ ] The web UI provides a download button on artifact rows and the artifact detail panel that triggers a browser download via the signed URL
- [ ] Downloading a `pending` artifact returns HTTP 409 with message `"artifact upload not confirmed"`
- [ ] Downloading an `expired` artifact returns HTTP 410 with message `"artifact has expired"`
- [ ] Downloading a non-existent artifact returns HTTP 404 with message `"artifact not found"`
- [ ] Downloading an artifact for a non-existent workflow run returns HTTP 404 with message `"run not found"`
- [ ] Downloading an artifact for a non-existent repository returns HTTP 404
- [ ] The `Content-Disposition` header on the blob download response uses `attachment; filename="<artifact-name>"` to trigger a save-as dialog in browsers
- [ ] The `Content-Type` header on the blob download response matches the stored `content_type` of the artifact record
- [ ] All clients (API, CLI, TUI, web UI) surface consistent error messages for non-downloadable artifacts

### Artifact Name Constraints

- [ ] Artifact names are case-sensitive strings between 1 and 255 characters
- [ ] Artifact names may contain alphanumeric characters, hyphens, underscores, dots, and forward slashes (to support path-like names such as `dist/bundle.js`)
- [ ] Artifact names must not contain `..` sequences (path traversal prevention)
- [ ] Artifact names must not start or end with whitespace
- [ ] Artifact names must be URL-safe when percent-encoded (the CLI and API use `encodeURIComponent` for the name path segment)
- [ ] Duplicate artifact names within a single workflow run are prevented at creation time; download always resolves to exactly one artifact

### Download URL Constraints

- [ ] Signed download URLs expire after 5 minutes (300,000 ms) from generation
- [ ] A download URL requested after the artifact's `expires_at` timestamp is not generated; a 410 is returned instead
- [ ] Signed URL tokens are HMAC-SHA256 over `key:expiresAt` using the server's blob signing secret
- [ ] Expired signed URL tokens return 403 from the blob download endpoint
- [ ] Invalid signed URL tokens return 403 from the blob download endpoint
- [ ] Each call to the download metadata endpoint generates a fresh signed URL (URLs are not cached or reused)

### Size and Content Constraints

- [ ] Artifacts up to 5 GB in size must be downloadable without error
- [ ] The `Content-Length` header on the blob download response matches the stored artifact `size` field
- [ ] Binary content types (e.g., `application/octet-stream`, `application/zip`, `application/gzip`) are streamed without transformation
- [ ] Text content types (e.g., `text/plain`, `application/json`, `text/xml`) are streamed without transformation or charset re-encoding
- [ ] Zero-byte artifacts (size = 0) return a 200 response with an empty body and correct headers

### Edge Cases

- [ ] Artifact whose blob has been deleted from storage (orphaned record): returns 404 with `"artifact data not found"` from the blob endpoint
- [ ] Artifact name containing URL-special characters (`%`, `+`, `#`, `?`, `&`): correctly resolved via percent-encoding in path
- [ ] Artifact name containing Unicode characters: correctly resolved via percent-encoding
- [ ] Concurrent download requests for the same artifact: each generates its own signed URL, both succeed independently
- [ ] Artifact expiry occurs between metadata fetch and blob download: the blob download URL was already signed and remains valid until its own expiry (the artifact record's expiry does not retroactively invalidate the signed URL within its 5-minute window)
- [ ] Network interruption during download: the client receives a partial file; the CLI should not overwrite a complete file with a partial one (write to temp then rename)
- [ ] Artifact attached to a release: download behavior is identical; release attachment metadata is informational only
- [ ] Repository transfer after artifact creation: artifact remains accessible under the new owner path because resolution is by repository ID internally
- [ ] Workflow run in `running` status with a confirmed artifact: artifact is downloadable (run completion is not a prerequisite)

## Design

### API Shape

**Endpoint:** `GET /api/repos/:owner/:repo/actions/runs/:id/artifacts/:name/download`

**Path Parameters:**
| Parameter | Type | Description |
|-----------|------|-------------|
| `owner` | string | Repository owner (user or organization) |
| `repo` | string | Repository name |
| `id` | integer | Workflow run ID (positive int64) |
| `name` | string | Artifact name (percent-encoded) |

**Response (200 OK):**
```json
{
  "id": 42,
  "repository_id": 7,
  "workflow_run_id": 123,
  "name": "coverage-report.html",
  "size": 245760,
  "content_type": "text/html",
  "status": "ready",
  "expires_at": "2026-04-21T12:00:00Z",
  "created_at": "2026-03-22T10:30:00Z",
  "updated_at": "2026-03-22T10:30:05Z",
  "confirmed_at": "2026-03-22T10:30:05Z",
  "release_tag": null,
  "release_asset_name": null,
  "release_attached_at": null,
  "download_url": "/api/blobs/download/repos%2F7%2Fruns%2F123%2Fartifacts%2F42%2Fcoverage-report.html?token=abc123def456&expires=1711234567890"
}
```

**Error Responses:**

| Status | Condition | Body |
|--------|-----------|------|
| 400 | Invalid run ID (non-numeric, negative, zero) | `{ "message": "invalid run id" }` |
| 400 | Missing artifact name | `{ "message": "artifact name is required" }` |
| 404 | Repository not found | `{ "message": "repository not found" }` |
| 404 | Workflow run not found | `{ "message": "run not found" }` |
| 404 | Artifact not found for the given run+name | `{ "message": "artifact not found" }` |
| 409 | Artifact exists but status is `pending` | `{ "message": "artifact upload not confirmed" }` |
| 410 | Artifact has expired (`expires_at` < now) | `{ "message": "artifact has expired" }` |

**Blob Download Endpoint:** `GET /api/blobs/download/:key`

This endpoint is a general-purpose signed blob download route. It:
1. Extracts `token` and `expires` from query parameters
2. Verifies the HMAC token against the blob store signing secret
3. Reads the blob from the local filesystem
4. Streams it with `Content-Type` from the stored metadata, `Content-Disposition: attachment; filename="<name>"`, and `Content-Length`

| Status | Condition | Body |
|--------|-----------|------|
| 200 | Valid token, blob exists | Raw bytes |
| 403 | Invalid or expired token | `{ "message": "forbidden" }` |
| 404 | Blob not found on disk | `{ "message": "not found" }` |

### CLI Command

**Command:** `codeplane artifact download <runId> <name>`

**Arguments:**
| Argument | Type | Required | Description |
|----------|------|----------|-------------|
| `runId` | number | yes | Workflow run ID |
| `name` | string | yes | Artifact name |

**Options:**
| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `--repo` | string | auto-detected from cwd | Repository in `OWNER/REPO` format |
| `--output` | string | artifact name | Output file path |

**Behavior:**
1. Resolve the repository reference from `--repo` or the current working directory
2. Call `GET /api/repos/:owner/:repo/actions/runs/:runId/artifacts/:name/download`
3. Parse the `download_url` from the response
4. Create parent directories for the output path if needed
5. Open a write stream to a temporary file (`.download` suffix) in the same directory as the target
6. Stream the response body from `download_url` to the temp file via Node.js `pipeline`
7. On successful completion, rename the temp file to the final output path
8. On failure, delete the temp file and report the error
9. Print structured output with `name`, `path`, `size`, and `content_type`

**Output (JSON mode):**
```json
{
  "name": "coverage-report.html",
  "path": "/home/user/coverage-report.html",
  "size": 245760,
  "content_type": "text/html"
}
```

**Error output:**
```
Error: artifact upload not confirmed (HTTP 409)
Error: artifact has expired (HTTP 410)
Error: artifact not found (HTTP 404)
Error: Failed to download artifact: 403 Forbidden
```

### Web UI Design

**Artifact Row Download Button:**
- Each artifact row in the workflow run artifacts list displays a download icon button (↓ arrow icon) on the right side of the row
- The button is only enabled when the artifact status is `ready`
- When status is `pending`, the button is disabled with tooltip "Upload not confirmed"
- When status is `expired` or `expires_at` is in the past, the button is disabled with tooltip "Artifact expired"
- Clicking the enabled button calls the download metadata endpoint, then opens `download_url` via `window.location.assign()` to trigger a browser save-as dialog

**Artifact Detail Panel Download:**
- The artifact detail panel (opened by clicking an artifact row) includes a "Download" action button
- Same enable/disable logic as the row button
- The button shows the artifact size in human-readable format (e.g., "Download (2.4 MB)")

**Progress and Feedback:**
- No in-browser progress bar is needed—the browser's native download manager handles progress
- On successful initiation, a brief toast notification shows "Download started: {name}"
- On error (409, 410, 404), a toast notification shows the error message from the API response

### TUI UI

**Download Action:**
- Pressing `D` (capital) on a focused artifact row initiates a download
- If the artifact status is `pending`, the status bar shows `"⚠ Artifact upload not confirmed"` in yellow for 3 seconds
- If the artifact status is `expired`, the status bar shows `"⚠ Artifact has expired"` in red for 3 seconds
- For `ready` artifacts, the TUI delegates to `codeplane artifact download <runId> <name>` as a child process
- The status bar shows `"↓ Downloading: {name}…"` while the download is in progress
- On success, the status bar shows `"✓ Downloaded: {name} → {path}"` in green for 5 seconds
- On failure, the status bar shows `"✗ Download failed: {error}"` in red for 5 seconds
- Multiple concurrent downloads are queued; the status bar shows `"↓ Downloading: {name}… (N queued)"`

### SDK Shape

The SDK exposes artifact download through the workflow service:

**`WorkflowService.getArtifactDownload(runId, name)`**
- Looks up the artifact record by `(workflowRunId, name)` using `getWorkflowArtifactByName`
- Validates the artifact status is `ready`
- Validates the artifact has not expired (`expires_at > now`)
- Generates a signed download URL via `blobStore.generateDownloadURL(artifact.gcsKey, DEFAULT_SIGNED_URL_EXPIRY_MS)`
- Returns the full artifact record plus the `download_url` field
- Returns appropriate error results for not-found, pending, and expired states

## Permissions & Security

### Authorization Roles

| Role | Can List Artifacts | Can Download Artifacts | Can Delete Artifacts |
|------|-------------------|----------------------|---------------------|
| Repository Owner | ✅ | ✅ | ✅ |
| Repository Admin | ✅ | ✅ | ✅ |
| Repository Write/Member | ✅ | ✅ | ✅ |
| Repository Read | ✅ | ✅ | ❌ |
| Anonymous (public repo) | ✅ | ✅ | ❌ |
| Anonymous (private repo) | ❌ | ❌ | ❌ |

- Download requires at minimum read access to the repository
- The signed download URL does not re-check repository permissions (it is token-authorized against the blob store signing secret). This means the 5-minute URL window is the security boundary—once a signed URL is issued, it is valid for anyone who possesses it until expiry.
- Signed URLs must never be logged in full in server access logs. Log only the blob key, not the token or expires parameters.

### Rate Limiting

| Endpoint | Limit | Window |
|----------|-------|--------|
| Download metadata (`/artifacts/:name/download`) | 60 requests per minute per user | Sliding window |
| Blob download (`/api/blobs/download/:key`) | 120 requests per minute per IP | Sliding window |
| Unauthenticated download metadata (public repos) | 30 requests per minute per IP | Sliding window |

Rate limit responses return HTTP 429 with `Retry-After` header.

### Data Privacy

- Artifact content may contain PII, secrets, or proprietary code. Signed download URLs must not be included in any server-side logs, analytics events, or error reports.
- The `gcs_key` (internal storage path) must never be exposed directly in API responses. The API response includes `download_url` which contains the key in encoded form, but this is gated by the HMAC token.
- Expired artifact blobs are hard-deleted from the filesystem by the cleanup scheduler. Once deleted, the data is unrecoverable.
- Artifact metadata (name, size, content type, timestamps) is retained in the database even after blob deletion until the artifact record itself is pruned.

### Funnel Metrics

| Metric | Definition | Success Target |
|--------|------------|----------------|
| **Download conversion rate** | `DownloadCompleted / DownloadRequested` | > 95% (failures should be rare for ready artifacts) |
| **Time to first download** | Median time from workflow run completion to first artifact download | Track and trend (no fixed target; lower is better) |
| **Expired artifact download attempts** | Count of 410 responses / total download requests | < 5% (users should not routinely hit expired artifacts) |
| **Pending artifact download attempts** | Count of 409 responses / total download requests | < 2% (pending artifacts should confirm quickly) |
| **Download throughput** | Total bytes served via artifact downloads per day | Track for capacity planning |

### Alerts

**Alert: High artifact download error rate**
- **Condition:** `rate(codeplane_artifact_download_errors_total{error_type!="not_found"}[5m]) > 0.1`
- **Severity:** Warning
- **Runbook:**
  1. Check `codeplane_artifact_download_errors_total` by `error_type` to identify the dominant error class
  2. If `blob_missing`: Run `SELECT id, gcs_key FROM workflow_artifacts WHERE status = 'ready' AND gcs_key NOT IN (...)` to identify orphaned records. Check cleanup scheduler logs for premature deletion. Verify blob store filesystem health (`df -h` on data dir).
  3. If `io_error`: Check disk I/O metrics, filesystem mount health, and Bun process file descriptor limits (`ulimit -n`). Look for `EMFILE` or `ENOSPC` in server logs.
  4. If `token_invalid`: Check if `CODEPLANE_BLOB_SIGNING_SECRET` was rotated without restarting all server instances. Verify clock sync across servers (NTP).
  5. If `pending`: Check workflow step execution pipeline for stalled artifact confirmations.

**Alert: Artifact download latency spike**
- **Condition:** `histogram_quantile(0.95, rate(codeplane_blob_download_duration_seconds_bucket[5m])) > 30`
- **Severity:** Warning
- **Runbook:**
  1. Check `codeplane_artifact_download_active` gauge for concurrent download count. High concurrency may indicate a traffic spike or runaway automation.
  2. Check disk I/O utilization on the blob store filesystem. Slow reads indicate disk saturation.
  3. Check `codeplane_blob_download_bytes_histogram` for a shift toward large artifacts.
  4. Check network throughput between server and clients. Verify no bandwidth throttling is active.
  5. If a single repository dominates, check for automated download loops (CI fetching artifacts in tight loops).

**Alert: Orphaned blob records**
- **Condition:** `increase(codeplane_artifact_download_errors_total{error_type="blob_missing"}[1h]) > 10`
- **Severity:** Critical
- **Runbook:**
  1. This indicates artifact records in the database reference blob keys that no longer exist on disk.
  2. Check cleanup scheduler logs for recent `sweepExpiredArtifacts` runs. Verify the scheduler is not deleting blobs for non-expired artifacts.
  3. Check if the blob store data directory was manually modified, moved, or corrupted.
  4. Check filesystem for recent `ENOSPC` errors that may have caused incomplete blob writes.
  5. For immediate mitigation, mark affected artifact records as `expired` so users see a clear status rather than a confusing 404.
  6. File an incident to investigate root cause of data loss.

**Alert: Signed URL clock skew**
- **Condition:** `increase(codeplane_artifact_download_errors_total{error_type="token_invalid"}[5m]) > 5`
- **Severity:** Critical
- **Runbook:**
  1. Signed URL validation depends on `Date.now()` matching between URL generation and verification.
  2. Check NTP sync status on all server instances: `timedatectl status` or `ntpq -p`.
  3. If multiple server instances exist, verify they share the same `CODEPLANE_BLOB_SIGNING_SECRET`. A secret mismatch between the API server and blob server causes all tokens to fail.
  4. Check if any recent deployment changed the signing secret without a coordinated restart.
  5. Immediate mitigation: restart all server instances to ensure consistent secret loading.

### Error Cases and Failure Modes

| Failure Mode | Detection | User Impact | Mitigation |
|-------------|-----------|-------------|------------|
| Artifact record not found | 404 from metadata endpoint | User cannot download | Verify artifact name, check if artifact was deleted |
| Artifact pending | 409 from metadata endpoint | User must wait | Retry after workflow step completes upload confirmation |
| Artifact expired | 410 from metadata endpoint | User cannot download | Re-run workflow to produce fresh artifact |
| Blob file missing (orphaned record) | 404 from blob endpoint | User gets download URL but blob fails | Alert triggers investigation; mark record as expired |
| Signed URL expired (>5 min) | 403 from blob endpoint | Browser download fails if user waited too long | Re-request download metadata for a fresh URL |
| Disk full | `ENOSPC` on blob read | Server 500 | Alert on disk usage; expand storage |
| File descriptor exhaustion | `EMFILE` on blob read | Server 500 | Alert on FD count; increase ulimit or reduce concurrency |
| Signing secret mismatch | 403 from blob endpoint on all requests | All downloads fail | Restart servers with consistent secret |
| Network timeout during stream | Client receives partial file | CLI writes partial file | CLI uses temp-file-then-rename pattern; user retries |
| Rate limited | 429 from either endpoint | User must wait | Respect `Retry-After` header |
